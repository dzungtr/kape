## 10. Open Questions

All questions below are resolved. No open items remain for v1 design.

---

### Q1: Multi-cluster support

**Decision:** One KAPE operator per cluster. Each cluster runs a fully self-contained KAPE instance — the operator reconciles CRDs, spawns handler pods, and writes Task records entirely within its own cluster. No hub operator, no kubeconfig federation.

Multi-cluster is treated as a **read-layer concern only**. All clusters write Task records to a shared audit DB (ClickHouse/PG). The UI Dashboard queries that shared DB with a `cluster` dimension on the Task schema, providing a cross-cluster view without any control-plane federation.

**CRD impact:** Add `cluster` field to the Task audit record. No changes to `KapeHandler`, `KapeTool`, or `KapeSchema`.

**Rationale:** A hub operator concentrates broad cluster credentials in one place (blast radius problem), creates a single point of failure across all clusters, and conflicts with per-cluster GitOps repo structure. Option B is operationally safer and architecturally cleaner.

### Q2: LLM cost management

**Decision:** Deferred entirely — not in v1, not in `KapeHandler` CRD. No operator involvement in cost management.

Spend visibility is a passive observability concern. OpenInference/OTEL traces emitted by handler pods already capture LLM call data. Engineers monitor spend via the observability stack. Revisit when there is a concrete cost problem to solve.

### Q3: Handler warm-up and replay

**Decision:** Add two fields to `KapeHandler.spec.trigger`:

```yaml
trigger:
  replayOnStartup: true # default: true
  maxEventAgeSeconds: 300 # default: 300 (5 minutes). 0 = no limit
```

**Behaviour:**

- `replayOnStartup: false` — implemented via NATS consumer deliver policy `DeliverNew`. Operator sets this on the NATS consumer at creation time. No backlog is replayed.
- `replayOnStartup: true` + `maxEventAgeSeconds` — operator uses `DeliverAll` or `DeliverByStartTime`. Handler runtime checks the CloudEvents `time` field against wall clock on every consumed message. Events older than `maxEventAgeSeconds` are ack'd immediately (not sent to DLQ — staleness is not an error) and a `StaleEventSkipped` Task record is written.

**Default behaviour** (300s window): recent backlog is processed on restart, old backlog is discarded with a full audit trail.

**Guidance:**

- Security handlers (Falco): set `maxEventAgeSeconds: 60` or `replayOnStartup: false`
- Cost/optimisation handlers (Karpenter): set longer windows (e.g. 3600) — actions are idempotent and opportunities may still be valid

**Implementation note:** Per-consumer age filtering is not natively supported by NATS JetStream at the consumer level — it is implemented in the handler runtime as a timestamp comparison before any LLM work begins. NATS deliver policy handles the `replayOnStartup` flag natively.

### Q4: Approval workflow UX

**Decision:** Full implementation deferred to v2.

**V1 contract:** Handler runtime supports `type: approval-request` as a valid action type in `actions[]`. When executed, the handler publishes to `kape.events.approvals.*` and stops — it does not wait for a response. Task record is written with `status: PendingApproval`.

**`KapeApproval` CRD structural contract (v2 implementation):**

```yaml
apiVersion: kape.io/v1alpha1
kind: KapeApproval
metadata:
  name: <auto-generated from task-id>
spec:
  taskRef: <task-id> # links to originating Task record
  decision: {} # copy of LLM decision awaiting approval
  requiredApprovers: 1
  approvers:
    - kind: SlackChannel
      ref: "#platform-oncall"
    - kind: Email
      ref: "platform@company.com"
  timeoutSeconds: 3600 # auto-reject if no response within window
  onTimeout: reject # reject | escalate
status:
  state: Pending # Pending | Approved | Rejected | TimedOut
  approvedBy: ""
  approvedAt: ""
  rejectedReason: ""
```

V2 operator watches `KapeApproval` objects. On approval, publishes `kape.events.approvals.approved` to NATS. A downstream handler picks it up to execute the deferred action.

### Q5: MCP server discovery

**Decision:** V1 is **manual `KapeTool` authoring only**. No auto-registration.

Rationale: Auto-registration would require MCP server workloads to hold write access to `kape.io` CRDs — this conflicts with the security model. Manual authoring enforces GitOps — a `KapeTool` CR is an explicit engineer decision that goes through version control.

**V2 auto-registration pattern:** Opt-in `kape-mcp-registrar` sidecar container added to the MCP server Deployment.

- Sidecar calls the MCP server's `capabilities` endpoint on startup
- Reads tool list and generates a `KapeTool` CR using an engineer-provided ConfigMap template
- Allow/deny filters and metadata remain engineer-controlled via the template — not fully automatic
- Sidecar watches for MCP server restarts and updates the CR if capabilities change

The sidecar is a separate KAPE component, not part of the operator. Engineers opt in explicitly.

### Q6: Cluster-wide configuration

**Decision:** No `KapeConfig` CRD. Cluster-wide configuration lives in a **ConfigMap** `kape-config` in `kape-system`. The operator Deployment reads it at startup. Helm chart `values.yaml` maps 1:1 to ConfigMap fields.

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: kape-config
  namespace: kape-system
data:
  llm.provider: anthropic
  llm.model: claude-sonnet-4-20250514

  embedding.model: text-embedding-3-small
  embedding.dimensions: "1536"

  auditDB.type: clickhouse
  auditDB.secretRef.name: kape-audit-db-credentials
  auditDB.secretRef.key: dsn

  dryRun: "false"

  nats.secretRef.name: kape-nats-credentials
  nats.secretRef.key: natsCredentials
```

**Design notes:**

- Credentials remain in Secrets — ConfigMap holds references only
- `llm` fields are cluster-wide defaults, overridable per `KapeHandler`
- `dryRun: true` is a global override — forces all handlers into dry-run regardless of per-handler setting
- `embedding` is cluster-wide — all Qdrant collections must use the same model and dimensions
- Operator injects relevant fields into handler pods as env vars at pod creation time

### Q7: KEDA scaling thresholds

**Decision:** Manual `natsLagThreshold` in `KapeHandler` only — for both v1 and v2. No auto-tuning. Removed from Future Work.

The operator never scrapes Prometheus or reads observability data. Metrics (consumer lag, LLM latency) are visible in the observability stack for engineers to observe and use to inform manual threshold adjustments via `KapeHandler` CRD updates (GitOps change).

Auto-tuning, if ever needed, belongs as a KEDA external scaler plugin — not inside the KAPE operator.

### Q8: Handler pod resource requests

**Decision:** Default resource requests/limits set in Helm chart, overridable per `KapeHandler` via `spec.resources`.

**Defaults:**

```yaml
resources:
  requests:
    cpu: 100m
    memory: 256Mi
  limits:
    cpu: 500m
    memory: 512Mi
```

**Rationale:** Handler pods are I/O-bound — wall-clock time is dominated by waiting on LLM API responses and MCP tool calls. CPU is only active during JSON parsing, Pydantic validation, and Jinja2 rendering. `100m` CPU request is sufficient for steady-state. `256Mi` memory covers Python process, LangGraph state, MCP tool registry, and in-flight event payload. CPU limit `500m` provides headroom for startup spikes (CRD loading, MCPToolkit initialisation).

**Interaction with KEDA:** KEDA scales on NATS consumer lag — entirely independent of CPU/memory utilisation. Resource requests exist solely for Kubernetes scheduler bin-packing. Low requests allow many handler pods to coexist on a node comfortably during scale-out events.

**Per-handler override:**

```yaml
spec:
  resources:
    requests:
      cpu: 100m
      memory: 256Mi
    limits:
      cpu: 500m
      memory: 512Mi
```
