## 1. Overview

KAPE is a polyglot monorepo at `github.com/kape-io/kape`. All components live in a single repository. A Go workspace (`go.work`) spans all Go modules for development convenience. Releases are driven by the `@changesets/cli` methodology — Docker images are built only at release time, not on every PR.

### Language summary

| Component      | Language   | Tooling                        |
| -------------- | ---------- | ------------------------------ |
| `operator`     | Go         | controller-runtime, go.work    |
| `task-service` | Go         | Chi router, pgx, go.work       |
| `adapters`     | Go         | go.work                        |
| `runtime`      | Python 3.12 | uv, LangGraph                 |
| `dashboard`    | TypeScript | React Router v7, Node.js       |
| `crds`         | YAML       | Generated by controller-gen    |
| `helm`         | Helm YAML  | Helm 3                         |
| `examples`     | YAML       | —                              |

## 3. Go Workspace

`go.work` at repo root spans six Go modules:

```
go 1.23

use (
    ./operator/domain
    ./operator/infra
    ./operator/controller
    ./operator/cmd
    ./task-service
    ./adapters
)
```

This allows `go test ./...` and `go build ./...` from the repo root during development without requiring published module versions for cross-module references.

## 4. Component: operator

The operator follows domain-driven design with dependency inversion. Four Go modules enforce strict layering — `domain` has zero external dependencies, `infra` adapts Kubernetes and external systems to domain interfaces, `controller` wires them together via thin reconcilers, `cmd` is the binary entry point.

### Dependency rule

```
cmd        →  controller, infra, domain
controller →  infra/ports (interfaces), domain (structs)
infra      →  domain (structs only)
domain     →  stdlib only
```

### Directory layout


### CRD generation

CRD YAML is generated from Go types in `infra/api/v1alpha1/` using `controller-gen`. Output is written to `crds/`. This is a build artifact — never edit `crds/` directly.

```bash
make generate
# runs: controller-gen rbac:roleName=kape-operator crd webhook \
#   paths=./operator/infra/... output:crd:artifacts:config=./crds
```

## 5. Component: task-service

Go REST API that persists Task records to PostgreSQL and serves the dashboard. The handler runtime never connects to PostgreSQL directly — all reads and writes go through this service.

```
task-service/
├── go.mod                        # github.com/kape-io/kape/task-service
├── package.json                  # { "name": "@kape-io/task-service", "version": "0.1.0" }
├── Dockerfile
├── cmd/task-service/
│   └── main.go
├── internal/
│   ├── api/                      # HTTP handlers (Chi router)
│   │   ├── tasks.go              # POST /tasks, PATCH /tasks/{id}, GET /tasks/{id}, DELETE /tasks/{id}
│   │   ├── handlers.go           # GET /handlers, GET /handlers/{name}
│   │   └── stream.go             # GET /tasks/stream (SSE)
│   ├── repository/               # pgx queries against tasks table
│   │   └── tasks.go
│   ├── model/                    # Task, TaskStatus, ActionResult types
│   │   └── task.go
│   └── stream/                   # SSE broadcaster
│       └── broadcaster.go
└── openapi/
    └── openapi.yaml              # OpenAPI spec — used to generate TypeScript types for dashboard
```

Three small Go binaries — each an HTTP server that normalises webhook/watch events into CloudEvents and publishes to NATS JetStream. All share a single Go module with common internal packages.

```

├── Dockerfile.falco
├── Dockerfile.alertmanager
├── Dockerfile.audit
├── cmd/
│   ├── falco/main.go             # Falco webhook receiver → CloudEvent → NATS
│   ├── alertmanager/main.go      # Alertmanager webhook receiver → CloudEvent → NATS
│   └── audit/main.go             # K8s audit webhook receiver → CloudEvent → NATS
└── internal/
    ├── cloudevents/              # shared CloudEvent builder
    │   └── builder.go
    └── nats/                     # shared NATS JetStream publisher
        └── publisher.go
```

## 8. Component: dashboard

TypeScript application using React Router v7 in framework mode. The Node.js server process is both the frontend server and the backend proxy to `kape-task-service`. No separate backend service.

```
dashboard/
├── package.json                  # React Router v7, Radix UI, Tailwind, remix-utils
├── tsconfig.json
├── vite.config.ts
├── Dockerfile                    # node:22-alpine, node server.js
└── app/
    ├── root.tsx
    ├── routes/
    │   ├── tasks._index.tsx      # Live task feed (SSE)
    │   ├── tasks.$id.tsx         # Task detail view
    │   ├── handlers._index.tsx   # Handler health overview
    │   └── handlers.$name.tsx    # Handler detail
    ├── components/
    │   ├── TaskTable.tsx
    │   ├── TaskStatusBadge.tsx
    │   ├── ActionTimeline.tsx
    │   └── HandlerCard.tsx
    └── types/
        └── generated/
```

TypeScript types for `Task`, `TaskStatus`, `ActionResult` are generated from `task-service/openapi/openapi.yaml` via `make generate` and committed to `dashboard/app/types/generated/`.

## 12. CI/CD Pipeline

### Workflow overview

```
.github/workflows/
├── ci.yaml           # PR: path-filtered lint → test → build binary per component
├── changesets.yaml   # push to main: open/update Version Packages PR
├── release.yaml      # Version Packages PR merged: git tag + docker build/push + helm publish
└── e2e.yaml          # PR + main: kind cluster smoke test
```

### PR workflow (`ci.yaml`)

Path-filtered — each component job only runs when files in its directory change. No Docker images are built on PRs.

| Component      | Steps                                              |
| -------------- | -------------------------------------------------- |
| `operator`     | golangci-lint → go test ./... → go build           |
| `task-service` | golangci-lint → go test ./... → go build           |
| `adapters`     | golangci-lint → go test ./... → go build           |
| `runtime`      | ruff check + ruff format --check → pytest          |
| `dashboard`    | eslint + tsc --noEmit → vite build                 |
| `helm`         | helm lint → helm template \| kubectl --dry-run     |

A changeset-bot GitHub App enforces that every PR targeting `main` includes at least one `.changeset/*.md` file.

### Release flow

```
1. Developer opens PR
   → runs: npx changeset
   → commits .changeset/<random-name>.md declaring bump type (patch|minor|major)
      and description per affected component
   → changeset-bot blocks merge until changeset present
   → ci.yaml runs lint/test/build for changed components

2. PR merges to main
   → changesets.yaml triggers
   → @changesets/action opens or updates "Version Packages" PR:
     - bumps version field in each affected component's package.json
     - appends entry to each component's CHANGELOG.md

3. Version Packages PR merged
   → release.yaml triggers
   → reads new version from each component's package.json
   → creates git tags: operator/v0.2.0, runtime/v0.1.1, etc.
   → docker build + push for each changed component:
       ghcr.io/kape-io/kape-operator:<version>
       ghcr.io/kape-io/kape-task-service:<version>
       ghcr.io/kape-io/kape-adapters-falco:<version>
       ghcr.io/kape-io/kape-adapters-alertmanager:<version>
       ghcr.io/kape-io/kape-adapters-audit:<version>
       ghcr.io/kape-io/kape-runtime:<version>
       ghcr.io/kape-io/kape-dashboard:<version>
   → helm package + push to ghcr.io OCI registry at helm chart version
```

### E2E smoke test (`e2e.yaml`)

Runs on every PR to `main` and on every push to `main`. Uses the latest released images (not PR build artifacts).

```
spin up kind cluster
  → install NATS JetStream + CloudNativePG
    → helm install kape (latest released images)
      → kubectl apply examples/
        → publish test CloudEvent to NATS
          → assert Task record reaches Completed status in task-service
```

## 13. Release Strategy

- **Versioning:** Semantic versioning per component. Components version independently — a fix to `runtime` does not bump `operator`.
- **Git tags:** Per-component tags in the form `<component>/v<semver>` (e.g., `operator/v0.3.0`).
- **Helm chart:** Chart version tracks its own semver in `helm/package.json`. The `appVersion` field in `Chart.yaml` reflects the operator version.
- **Docker registry:** `ghcr.io/kape-io/` — all images in the same GitHub Container Registry org.

## 14. Decision Record

| Decision | Choice | Rationale |
| --- | --- | --- |
| Monorepo vs multi-repo | Monorepo | Single PR spans all components; CRD generation, type sharing, and e2e tests are simpler |
| Go module structure | go.work workspace (6 modules) | Preserves operator's domain/infra/controller layering from spec; independent versioning per Go component |
| Adapter language | Go | Shared NATS client + CloudEvents builder with task-service; static binaries; consistent with operator |
| CRD generation | controller-gen from Go types | Single source of truth in `operator/infra/api/v1alpha1/`; prevents schema drift |
| Python packaging | uv | Faster CI installs than poetry; deterministic lock file; idiomatic for new projects in 2026 |
| CI system | GitHub Actions | Native to GitHub; path-filtered per component |
| Release methodology | @changesets/cli | Per-component independent versioning; automated CHANGELOG; Docker images built only at release |
| Subfolder naming | No `kape-` prefix | Repo name (`kape`) already provides context; cleaner paths |
